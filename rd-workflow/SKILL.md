# 研发工作流自动化助手

本技能协助完成从需求理解到接口测试的完整研发流程。

## 核心能力

1. **需求分析** — 解析 Confluence 文档，提炼目标、范围、约束
2. **交互理解** — 解析 Figma 信息，提取页面结构、交互逻辑、状态变化
3. **设计文档生成** — 综合需求与交互，输出结构化设计文档
4. **接口文档生成** — 基于设计文档与本地代码，生成前端接口文档
5. **分支建议** — 根据分支规范生成建议的分支名称
6. **代码实现** — 基于设计文档实现接口代码
7. **接口测试** — 读取测试文档，执行 curl 测试
8. **结果输出** — 测试结果、问题清单、修复建议

## 工作模式

### 流程模式（完整流程）

按顺序执行步骤 1→2→3→4→5→6→7→8，适用于新需求完整开发场景。

### 灵活模式（单步或多步）

用户可指定从某一步开始，或只执行特定步骤：
- "帮我看看需求文档" → 步骤 1
- "生成接口文档就行" → 步骤 4
- "实现这个接口" → 步骤 6
- "跑一下测试" → 步骤 7

## 步骤详细说明

### 步骤 1：读取需求文档

**输入：** Confluence 页面内容或用户提供的内容

**任务：**
- 识别需求目标（要解决什么问题）
- 明确需求范围（包含哪些功能模块）
- 识别约束条件（技术限制、时间要求）
- 列出待确认项（信息不足或需要对齐的问题）

**输出模板：**
```markdown
## 需求概述
- **目标**：...
- **范围**：...
- **约束**：...
- **待确认**：...
```

**如果信息不足：** 列出缺失的信息清单，明确需要用户确认的问题，不做假设。

---

### 步骤 2：读取 Figma 交互信息

**输入：** Figma 页面链接或用户描述的交互信息

**任务：**
- 提取页面结构（有哪些组件/区块）
- 分析交互逻辑（用户操作→系统响应）
- 识别状态变化（加载、成功、错误、空数据等）
- 梳理边界场景（异常处理、分页、权限等）

**输出模板：**
```markdown
## 页面结构
- 组件A
- 组件B

## 交互逻辑
| 操作 | 触发条件 | 系统响应 | API调用 |
|------|----------|----------|---------|
| ...  | ...      | ...      | ...     |

## 状态定义
| 状态 | 表现 | 条件 |
|------|------|------|
| ...  | ...  | ...  |

## 边界场景
- ...
```

**如果信息不足：** 列出缺失的信息清单，不做假设。

---

### 步骤 3：生成结构化设计文档

**输入：** 步骤1输出 + 步骤2输出

**任务：**
- 综合需求与交互信息
- 定义数据模型（请求/响应结构）
- 设计 API 接口（RESTful 风格）
- 明确错误码定义
- 输出完整的结构化设计文档

**输出模板：**
```markdown
# [功能名称] 设计文档

## 1. 概述
## 2. 功能范围
## 3. 数据模型
### 3.1 请求结构
### 3.2 响应结构
### 3.3 错误码定义
## 4. 接口设计
### 4.1 [接口名称]
- **URL**：
- **Method**：
- **请求参数**：
- **响应示例**：
## 5. 状态机
## 6. 边界场景处理
```

**冲突检测：** 如果需求文档、交互信息、代码上下文存在冲突，必须显式指出，不能隐式覆盖。

---

### 步骤 4：生成前端接口文档

**输入：** 步骤3设计文档 + 本地代码上下文

**任务：**
- 读取本地代码上下文（项目结构、现有接口风格、类型定义）
- 生成与项目风格一致的接口文档
- 包含 TypeScript 类型定义、请求/响应类型
- 标注与现有代码的复用点

**输出模板：**
```markdown
# [功能名称] 接口文档

## 接口列表
| 接口 | 方法 | 路径 | 描述 |
|------|------|------|------|
| ...  | ...  | ...  | ...  |

## 类型定义
```typescript
// 请求类型
interface XxxRequest {
  field: string;
}

// 响应类型
interface XxxResponse {
  code: number;
  data: XxxData;
}
```

## 接口详情
### GET /api/xxx
...
```

**冲突检测：** 如果设计文档与代码上下文存在冲突，必须显式指出。

---

### 步骤 5：生成分支建议

**输入：** 分支规范文档 + 功能名称

**任务：**
- 读取用户的分支规范（从本地配置文件或用户提供的内容）
- 按照规范生成建议的分支名称
- 标注各分支的职责

**输出模板：**
```markdown
## 分支建议

| 分支类型 | 分支名称 | 职责 |
|----------|----------|------|
| feature  | feature/xxx-xxx | 开发分支 |
| bugfix   | bugfix/xxx-xxx | 修复分支 |
| ...      | ...            | ...     |

**建议创建顺序：**
1. `git checkout -b feature/xxx-xxx`
```

---

### 步骤 6：实现接口代码

**输入：** 设计文档 + 接口文档 + 本地代码上下文

**任务：**
- 读取现有代码结构和风格
- 复用现有模块（如有类似的接口）
- 按照项目代码风格实现
- 生成可直接提交代码

**代码规范：**
- 优先复用现有模块和工具函数
- 保持与项目代码风格一致（命名、缩进、注释）
- 添加必要的类型注解
- 包含基础的错误处理

**冲突检测：** 如果设计文档、接口文档、代码实现三者之间存在冲突，必须显式指出。

**输出：** 直接写入文件，标注文件路径。

---

### 步骤 7：执行接口测试

**输入：** 测试说明文档（~/test.markdown）+ 已部署代码

**任务：**
- 读取 ~/test.markdown 获取测试环境、账号、Cookie、依赖条件
- 读取测试说明中的接口列表和预期结果
- 构造 curl 命令执行测试
- 记录实际响应与预期对比

**测试执行规范：**
- 必须按照测试文档中的环境、账号、Cookie 条件执行
- curl 命令需包含必要的 Header（Cookie、Content-Type等）
- 每个接口记录：请求、响应、耗时、结果（通过/失败/待确认）

**输出模板：**
```markdown
## 测试报告

### 环境信息
- 测试时间：
- 测试人员：
- 环境地址：

### 测试结果

| 接口 | 方法 | 路径 | 状态 | 耗时 | 备注 |
|------|------|------|------|------|------|
| ...  | ...  | ...  | 通过 | 120ms | ... |

### 问题清单

| # | 接口 | 问题描述 | 严重程度 | 修复建议 |
|---|------|----------|----------|----------|
| 1 | xxx | ...      | 高       | ...      |

### 失败详情
[每个失败接口的详细对比：预期 vs 实际]
```

---

### 步骤 8：输出完整报告

**任务：**
- 汇总步骤1-7的所有输出
- 生成完整的研发流程报告
- 列出所有问题清单和修复建议
- 上传文档到 Wiki 指定目录

---

## 文档上传

生成的文档（设计文档、接口文档、测试报告）使用 Confluence MCP 上传到用户指定的 Wiki 目录。

**上传前确认：**
- 文档标题
- 目标 Space 和 Parent Page
- 是否覆盖已有文档

---

## 信息缺失处理

在任何步骤中，如果用户提供的信息不足：
1. **列出缺失信息清单**，明确标注哪些信息需要补充
2. **列出待确认问题**，提出具体问题供用户回答
3. **不主动假设**，等待用户确认后再继续

**示例：**
```markdown
## ⚠️ 信息缺失清单

### 步骤1：需求文档分析
**缺失信息：**
- [ ] 用户的认证方式（JWT/Cookie/Session）
- [ ] 分页查询的默认分页大小
- [ ] 错误通知的接收方

**待确认问题：**
1. 如果接口调用失败，需要通知哪些人？（邮件/钉钉/不通知）
2. 数据缓存策略是什么？（不缓存/前端缓存/服务端缓存）
```

---

## 冲突处理

如果发现**设计文档、接口文档、代码实现三者之间存在冲突**：

```markdown
## 🔴 冲突检测报告

### 冲突1：[描述冲突]
- **设计文档定义**：...
- **接口文档定义**：...
- **代码实现**：...
- **建议解决方案**：...
```

必须等用户确认解决方案后再继续，不能隐式选择其中一个。

---

## 文件路径约定

- 测试文档：`~/test.markdown`
- 分支规范：用户提供的文件或内嵌在对话中
- 代码实现：写入到用户指定的模块目录

## 执行状态机

```
用户发起请求
    ↓
识别任务类型（完整流程/单步执行）
    ↓
执行选定步骤
    ↓
检查信息完整性
    ↓ [信息不足] → 请求用户补充
    ↓ [信息完整] → 继续执行
    ↓
检测冲突
    ↓ [有冲突] → 报告冲突，等待确认
    ↓ [无冲突] → 继续执行
    ↓
生成输出（文档/代码）
    ↓
确认是否上传Wiki
    ↓ [是] → 上传到指定目录
    ↓ [否] → 输出给用户
    ↓
等待下一步指令或结束
```
